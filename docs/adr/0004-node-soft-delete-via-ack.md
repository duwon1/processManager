# ADR-0004: 노드 소프트 삭제 — 언인스톨 ACK 기반 최종 삭제

- **상태**: 채택
- **관련 코드**: `NodeService`(deleteNode / completeUninstall / completeUninstallOnDisconnect), `AgentLifecycleWebSocketController`, `DeletedNodesMapper`

## 맥락

사용자가 노드를 삭제하면 원격 서버의 에이전트도 실제로 제거(언인스톨)되어야 한다.
단순히 DB 행만 지우면 다음 문제가 생긴다.

- 에이전트는 여전히 살아서 재접속하고, 노드가 **유령처럼 되살아난다**.
- 삭제 시점에 노드가 오프라인이면 언인스톨 명령을 전달할 수 없다.
- 구버전 에이전트는 언인스톨 완료 ACK를 보내지 않는다.

## 결정

즉시 삭제 대신 **삭제 대기(soft delete) → 확인 후 최종 삭제** 2단계로 처리한다.

1. `deleteNode`: 노드를 `status="D"`(삭제 대기)로 전환하고 `deleted_nodes`에 예약을 남긴다.
   - **온라인**이면 즉시 언인스톨 명령을 전송한다.
   - **오프라인**이면 목록에서 바로 제거하되 예약만 남겨, 다음 접속 때 자가 삭제 명령을 재전송한다.
2. 에이전트가 언인스톨을 마치면 `agent.uninstall-ack`를 보내고 → `completeUninstall`에서 최종 삭제한다.
3. **폴백**: ACK를 안 보내는 구버전은 (a) DISCONNECT를 완료 신호로 간주하거나
   (b) 짧은 유예(`LEGACY_UNINSTALL_GRACE` = 5초) 후 정리한다.
4. 삭제 대기 노드는 heartbeat·재연결로 다시 온라인으로 되살아나지 않게 상태를 고정한다.

## 결과

**장점**
- 원격 에이전트의 실제 제거를 확인한 뒤에만 레코드를 지워 "유령 노드"를 방지한다.
- 오프라인·구버전 등 엣지 케이스에서도 서버 목록이 무한 대기하지 않는다.

**단점 / 비용**
- 삭제가 비동기 2단계라 상태 머신(D 상태, 예약 테이블, 유예 스케줄)이 복잡하다.
- ACK와 DISCONNECT 폴백이 겹칠 수 있어 최종 삭제는 **멱등**하게 구현해야 한다.
- 삭제 예약(`deleted_nodes`)이라는 별도 저장소를 유지·정리해야 한다.
