# Process Manager Project Progress

작성일: 2026-05-16

## 요약

현재 프로젝트는 **웹에서 원격 노드를 등록하고, 실시간 모니터링/관리하고, 팀 단위로 공유하는 MVP 후반 단계**입니다.

대략적인 진행도는 다음처럼 보는 것이 맞습니다.

| 영역 | 상태 | 진행도 |
|---|---|---:|
| 기본 인증/배포 | 운영 사용 가능 | 80% |
| Linux 에이전트 관리 | 핵심 기능 구현됨 | 80% |
| 팀/권한/공유 | 주요 흐름 구현됨 | 70% |
| 설치/삭제/업데이트 생명주기 | 구현됨, 실기기 검증 필요 | 70% |
| Windows 에이전트 | 설치/모니터링/삭제 중심 구현 | 45% |
| 모바일 UI/UX | 많이 개선됐지만 추가 조정 필요 | 60% |
| 테스트/운영 안정성 | 수동 검증 중심 | 45% |

전체적으로는 **포트폴리오용 베타 서비스 수준까지는 올라왔고, 실제 사용 안정성 검증과 Windows 기능 확장이 다음 핵심 작업**입니다.

- React 라우팅: `/login`, `/main`, `/teams`, `/dashboard/:nodeId`
- Bootstrap, Bootstrap Icons, Bootswatch Vapor 기반 UI
- 공통 Toast, Confirm Dialog 컴포넌트
- GitHub Actions 기반 Fly.io 자동 배포
- Fly.io 앱 `procmanager` 배포 구조

## 현재 리스크

- Windows 기능은 아직 Linux 대비 제한적임
- Windows updater 구현과 capability 표시가 불일치할 수 있음
- 팀 권한은 구현되어 있으나 전체 화면/전체 WebSocket 경로에 대한 회귀 테스트가 부족함
- 모바일 UI는 개선됐지만 실제 기기별 검증이 더 필요함
- 운영 DB 백업/복구/마이그레이션 절차가 문서화되어 있지 않음
- 설치 스크립트는 실제 사용자 환경의 PowerShell 정책, 백신, 네트워크 상태에 영향을 받을 수 있음

## 결론

현재 프로젝트는 **혼자 쓰는 포트폴리오용 원격 서버 관리 도구로는 핵심 기능이 대부분 갖춰진 상태**입니다.

다음 단계는 새 기능을 크게 늘리기보다, 먼저 **Windows 실기기 검증, Linux 회귀 검증, 팀 권한 검증**을 끝내는 것이 좋습니다. 그 다음 Windows 기능을 Linux 수준으로 맞추면 프로젝트 완성도가 크게 올라갑니다.
